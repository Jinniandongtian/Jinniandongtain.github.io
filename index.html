如何定位慢查询？

  1. 使用工具Skywalking
  2. 开启MySQL自带的慢查询日志，设置一个阈值，一旦某个接口查询的时间超过阈值就会记录到日志中

如果SQL语句执行很慢，如何分析优化呢？

  1. 使用exlplain语句查询执行计划
  2. 通过key和key_len检查是否走了索引
  3. 通过type字段查看是否有进一步优化的空间
  4. 通过extra判断是否出现了回表的情况

什么是索引？索引的底层数据结构了解吗？

索引是帮助MySQL高效获取数据的数据结构，主要是用来提高数据库的检索效率，降低数据库的IO成本，同时通过索引对数据进行排序，降低数据排序的成本（创建索引时就需要排序）。

选择B+树的原因：

  1. 阶数更多，路径更短
  2. 只有叶子结点存储数据，查找效率更加稳定
  3. 方便范围查找

聚簇索引和非聚簇索引 以及 什么是回表查询？

  1. 聚簇索引：数据和索引结构放在一块，B+树的叶子节点保存了整行数据，有且只有一个
  2. 非聚簇索引：数据和索引分开存储，B+树的叶子节点保存对应的主键，可有多个
  3. 回表查询：通过二级索引找到对应的主键值，再到聚簇索引中查找整行数据，就是回表

覆盖索引

  1. 覆盖索引是指查询使用了索引，所返回的列，必须在索引中全部能够找到

  2. MySQL超大分页怎么处理？提升查询效率，就用覆盖索引，在数据量比较大时，limit分页查询，需要对数据进行排序，效率低

    解决方案：覆盖索引+子查询

索引创建的原则有哪些？

  1. 数据量比较大，且查询比较频繁的表(重要)
  2. 常用作查询条件、排序、分组的字段（重要）
  3. 尽量使用联合索引（重要）
  4. 要控制索引的数量（重要）
  5. 使用NOT NULL去约束字段
  6. 使用区分度高的字段
  7. 字段的长度较长时，可以针对字段的特点，建立前缀索引

导致索引失效的情况

  1. 违反最左前缀法则
  2. 范围查询右边的列，不能使用索引
  3. 在索引上进行运算操作会导致索引失效
  4. 字符串不加单引号
  5. 模糊查询有可能导致索引失效

SQL语句的优化

  1. 避免使用select *
  2. 避免导致索引失效的写法
    • 违反最左前缀匹配原则
    • 范围查询右边的列索引会失效
  3. 尽量用union all 代替 union，因为union会将查询结果进行过滤
  4. 避免在where子句中对字段进行表达式操作
  5. Join优化，能用inner join就不用left join和right join，INNER JOIN 只返回匹配的行，而 LEFT JOIN 则返回左表的所有行，不匹配的行则设置为空

并发事务问题

  1. 脏读：一个事务读到了另一个事务没有提交的数据
  2. 不可重复读：在一个事务内部两次读取到的数据不一样
  3. 幻读：一个事务查询数据时没有对应的数据行，但是在插入数据时，又发现这行数据已经存在，好像出现了“幻影”

MySQL日志

  1. redo log （重做日志）：物理日志，用于数据持久化
  2. undo log：逻辑日志，提供回滚和MVCC
  3. redo log保证了数据的持久性，undo log保证了数据的原子性和一致性

分库分表

  1. 垂直分库：以表为依据，根据业务将不同表拆分到不同库中
  2. 垂直分表：将表中字段分到不同表中
  3. 水平分库：将一个库的数据拆分到多个库中
  4. 水平分表：将一个表的数据分散到多个表中
  5. 分库会出现的问题：
    • 分布式事务一致性问题
    • 主键避重
    • 跨节点关联查询

Redis

如何解决缓存穿透问题？

  1. 方法1：缓存空值
  2. 方法2：使用布隆过滤器：用于检索一个元素是否在一个集合中。根据Id，用3个hash函数计算下标存入bitmap数组中。判断id是否存在时，就看计算出的3个位置是否都为1

如何解决缓存击穿问题？

  1. 方法1：使用逻辑过期（高可用，性能优秀，但是不能保证数据一致性）：设置一个过期时间字段，如果查询到数据过期，就新开一个线程，新线程获取锁，查询数据库并更新缓存。
  2. 方法2：使用互斥锁，当线程查询缓存未命中时都会获得锁，然后查询数据库更新缓存

如何解决缓存雪崩的问题？

缓存雪崩：短时间内大量key过期或者redis宕机，导致大量请求到达DB

  1. 设置过期时间TTL为随机值
  2. 使用Redis集群提高服务的可用性
  3. 给缓存业务添加降级限流策略（保底策略）
  4. 给业务添加多级缓存

如何解决双写一致性的问题？

强一致性

  1. 延迟双删
  2. 加分布式锁：使用共享锁，读不互斥，写互斥

非强一致性

  1. 使用MQ消息队列：修改数据时，先写入数据库，再发送一条消息给数据库，redis端监听消息
  2. 使用binlog日志，canal实现

5-10面试内容准备

AQS

  • 多线程中的抽象队列同步器，是一种锁机制，作为一种基础框架使用的，例如ReentrantLock、Semaphored、countdownlatch都是基于AQS实现的。
  • AQS内部维护了一个先进先出的双向队列，队列中存储的是排队的线程
  • AQS内部还有一个属性state，默认为0。当队列中的一个线程修改state为1，则默认当前线程获取了锁
  • 为保证对state修改的原子性，使用的是CAS操作。
  • 补充：AQS既能实现公平锁（排在队头的元素先获取锁）也能实现非公平锁（谁抢到算谁的）

ReentrantLock

  • 可重入锁，调用Lock方法获取了锁之后，再次调用Lock方法不会再阻塞。
  • ReentrantLock主要利用AQS+CAS实现的。
  • 支持公平锁和非公平锁，无参构造方法默认是非公平锁，也可以传参设置为公平锁

Sychronized和lock的区别

  • 语法层面：sychronized是关键字，lock是接口，jdk提供。退出同步代码块时，sychronized会自动释放；而使用lock需要手动调用unlock方法释放锁。
  • 功能层面：lock接口提供了公平锁、可中断、可超时、多条件变量等功能。
  • 性能层面：在竞争激烈时，Lock的实现通常会具体有更好的性能。

导致并发程序出现问题的根本原因是什么？

  1. 原子性：比如说抢票问题，ticknum–，但是–操作不是一个原子性操作，在底层实现中会先获取ticknum的值，但是如果此时线程阻塞，其他线程获取到值并修改就会出现问题。例如A线程获得ticknum=10,阻塞，B线程获取ticketnum=10并减1为9，此时A执行ticknum–，还是9。ticknum从10->9卖出了两次票。

    解决方法：sychronized或lock。

  2. 有序性问题：发生了指令重排序。解决：volatile关键字

  3. 可见性问题：JIT会优化代码。解决：volatile关键字

线程池中有哪些常见的阻塞队列

  • LinkedBlockingQueue : 默认无界，是懒惰的，创建节点的时候添加数据，有两把锁
  • ArrayBlockingQueue : 有界，提前初始化Node数组，有一把锁

如何确定核心线程数

    • 计算密集型：N+1
